#### 本步涉及的技术与设计

| 技术/设计 | What | Why | Which | How |

| **uv** | Astral 出品的现代化 Python 包管理器，替代 pip + venv + pip-tools | 比传统 pip 快 10-100 倍，自动管理虚拟环境，统一依赖解析 | pip+venv（慢、手动） vs poetry（较重） vs uv（快、轻、现代），选 uv | `uv init` 创建项目，`uv add` 添加依赖，`uv run` 运行命令 |
| **pyproject.toml** | Python 项目标准化配置文件，声明项目元数据和依赖 | Python 社区推荐标准（PEP 621），替代 setup.py 和 requirements.txt | setup.py（过时） vs requirements.txt（无元数据） vs pyproject.toml（标准），选 pyproject.toml | 在文件中声明 `[project]` 和 `[tool.uv]` 段 |
| **.gitignore** | Git 忽略规则文件 | 防止敏感信息（.env）和运行时数据（data/）进入版本库 | 必须使用，无替代方案 | 模板来自 gitignore.io，针对 Python 项目定制 |
| **.env + .env.example** | 环境变量文件模式 | API Key 等敏感信息不入版本库，.env.example 提供模板供他人参考 | 硬编码（不安全） vs 配置文件（可见） vs .env（安全+便捷），选 .env | python-dotenv 加载，.env.example 列出所有需要的变量 |

| **Pydantic 模型** | Python 数据校验库，用 class 定义数据结构并自动校验 | 强类型约束、自动序列化/反序列化、IDE 补全友好，企业级项目标配 | dataclass（无校验） vs attrs（需额外学习） vs Pydantic（功能最全），选 Pydantic | 定义 BaseModel 子类，声明字段类型和默认值 |
| **ABC 抽象基类** | Python 的 abc 模块，定义抽象接口 | 强制子类实现所有方法，保证存储后端接口一致性 | 鸭子类型（无强制） vs Protocol（较新） vs ABC（成熟标准），选 ABC | 定义 StorageBackend(ABC)，所有方法加 @abstractmethod |

| **Rich** | Python 终端富文本渲染库 | 流式输出需要实时渲染、表格展示需要格式化、Markdown 渲染 | print（无样式） vs click（功能有限） vs rich（功能最全），选 rich | 用 rich.console.Console 进行输出，rich.markdown 渲染 |
| **prompt_toolkit** | 高级命令行交互库 | 支持输入历史、自动补全、多行输入，比 input() 强大得多 | input()（最基础） vs prompt_toolkit（全功能），选 prompt_toolkit | 用 prompt() 替代 input()，配置历史和补全 |

#### 本步涉及的技术与设计

| **aiosqlite** | SQLite 的异步 Python 驱动 | 项目全链路异步，同步 sqlite3 会阻塞事件循环 | sqlite3（阻塞） vs aiosqlite（异步轻量） vs SQLAlchemy async（重量级 ORM），选 aiosqlite | `async with aiosqlite.connect(path)` 建立连接 |
| **工厂模式** | 设计模式，根据配置创建不同的存储后端实例 | 解耦后端选择与业务代码，新增后端无需修改业务层 | if-else（硬编码） vs 工厂模式（灵活可扩展），选工厂模式 | StorageFactory.create(config) 返回 StorageBackend 实例 |
| **数据库迁移** | 管理数据库表结构的变更 | 企业级项目的标准做法，保证表结构版本可控 | 手动建表（不可追溯） vs 迁移脚本（可追溯可回滚），选迁移脚本 | init_db.py 中按顺序执行 CREATE TABLE IF NOT EXISTS |

### Step 6：对话引擎（LLM 调用 + Memory + 流式 + 超时重试 + Token 统计）

| 项目 | 内容 |
|------|------|
| **目标** | 实现完整的对话引擎，这是整个项目的核心模块 |
| **涉及文件** | `src/core/chat_engine.py` |

| **Git** | commit + tag `step-6-chat-engine` |

#### 本步涉及的技术与设计

| 技术/设计 | What | Why | Which | How |
|-----------|------|-----|-------|-----|
| **ChatOpenAI (langchain-openai)** | LangChain 的 OpenAI 兼容聊天模型封装 | 兼容所有 OpenAI API 格式的提供商（OpenAI、DeepSeek、Ollama 等），配置 api_base 即可切换 | langchain 直接调用（无统一接口） vs ChatOpenAI（标准化封装），选 ChatOpenAI | `ChatOpenAI(base_url=..., api_key=..., model=..., streaming=True)` |
| **ConversationBufferMemory** | LangChain 的会话记忆组件 | 自动维护消息历史列表，每轮自动传入上下文 | 手动维护列表（易出错） vs Memory 组件（自动管理），选 Memory | 初始化 Memory，绑定到 Chain，自动追加每轮消息 |
| **astream** | LangChain 的异步流式输出 API | 逐 token 输出，用户体验远好于等待完整响应 | invoke（等待完整响应） vs astream（逐 token），选 astream | `async for chunk in chain.astream(...)` |
| **with_retry** | 重试装饰器/配置 | LLM API 调用可能因网络/限流失败，需要自动重试 | 手动 try-except（重复代码） vs with_retry（声明式配置），选 with_retry | ChatOpenAI(max_retries=3, timeout=30) |

#### 本步涉及的技术与设计

| 技术/设计 | What | Why | Which | How |
|-----------|------|-----|-------|-----|
| **环境覆盖合并** | 基础配置 + 环境特定配置合并的机制 | 不同环境共享大部分配置，只少数项不同，合并机制避免重复 | 每环境一份完整配置（冗余） vs 覆盖合并（DRY），选覆盖合并 | config_manager 先加载 `config.yaml`，再用 `config.{APP_ENV}.yaml` 深度覆盖 |
| **APP_ENV 环境变量** | 指定当前运行环境的标准开关 | 环境切换不应改代码，环境变量是 12-Factor App 标准做法 | 启动参数 vs 配置文件标记 vs 环境变量，选环境变量 | 程序启动时读取 `APP_ENV`，默认 dev |
| **多 .env 文件** | 每环境独立的敏感信息文件 | 不同环境用不同密钥/数据库密码，必须隔离 | 单 .env 轮换（易错） vs 多 .env（清晰），选多 .env | `.env.dev`/`.env.test`/`.env.prod`，按 APP_ENV 加载对应文件 |

#### 实现要点

1. **配置合并策略**：`最终配置 = config.yaml（基础） + config.{env}.yaml（覆盖）`，覆盖配置只写"与基础不同的字段"。
2. **环境隔离**：每环境的 `.env` 独立，存储后端独立（dev=SQLite、prod=MySQL、test=独立测试库），密钥独立。
3. **切换方式**：`APP_ENV=dev uv run python src/main.py`，无需改动任何代码或配置。

#### 验收标准（详见 REQ-001）

- [ ] dev：SQLite + 开发模型，建用户→对话→导出 全通过
- [ ] test：独立测试库，`uv run pytest` 全绿
- [ ] prod：MySQL + 正式模型（或模拟），核心功能冒烟通过
- [ ] 三环境密钥隔离、数据库隔离
