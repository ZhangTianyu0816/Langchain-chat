> **用途**：登记所有「不在当前开发阶段实现，但最终必须交付」的需求。
> **原则**：需求不进代码，但必须有可追溯的书面登记，防止口头约定最后被遗忘。
> **维护规则**：每条需求一个登记项，含编号、来源、状态、描述、预留扩展点、计划实现步骤、验收标准。

### 需求描述

支持 **dev（开发）/ test（测试）/ prod（生产）三种运行环境**，实现配置与敏感信息的彻底隔离：

1. **三套独立配置**：每种环境有独立的业务配置（默认模型、存储后端类型、超时重试参数等）。
2. **三套独立敏感信息**：每种环境有独立的 API Key、数据库密码等（存入各自的 `.env`，互不污染）。
3. **运行时一键切换**：通过环境变量（如 `APP_ENV=dev|test|prod`）或启动参数切换环境，无需改动代码。
4. **配置隔离验证**：切换环境后，加载的配置与数据源必须真正隔离（如 dev 的数据库不会被 prod 误写，dev 的密钥不会泄漏到 prod）。

### 为什么不在当前阶段（Step 1~14）实现

- 遵循 **YAGNI 原则**（You Aren't Gonna Need It，不过度设计）：当前阶段只有一个运行环境，引入多环境会增加复杂度、拖慢逐step验证节奏。
- 这是「**运行/部署层面**」的增强，不依赖任何业务功能，放最后实现最合理，且能反向检验「配置与代码解耦」做得是否到位。

### 预留的扩展点（Step 1~14 落地）

1. `config.yaml` 采用清晰的分层结构，字段命名按「将来可被环境覆盖」的方式组织。
2. 当前阶段为单一环境运行，但 `app.env` 字段以注释形式预留。
3. 配置加载代码（Step 2 的 `config_manager`）后续支持「基础配置 + 环境覆盖配置」的合并机制即可，无需重构数据结构。

### 计划实现步骤

**Step 15：多环境配置区分实现与验证**（新增于《实施步骤计划》v1.2，详见该文档）

- 引入环境区分机制：`APP_ENV` 环境变量 + `config.{env}.yaml` 覆盖配置 + `.env.{env}` 敏感值
- 改造配置加载逻辑：基础配置 + 环境覆盖合并
- 三环境分别配置并冒烟验证

### 验收标准（必须全部满足）

- [ ] 设置 `APP_ENV=dev` 时，程序加载 dev 配置（SQLite + 开发用便宜模型），冒烟测试通过：创建用户 → 发起对话 → 导出会话
- [ ] 设置 `APP_ENV=test` 时，程序加载 test 配置（独立测试数据库），`uv run pytest` 全部通过
- [ ] 设置 `APP_ENV=prod` 时，程序加载 prod 配置（MySQL + 正式模型或模拟），核心功能冒烟通过
- [ ] 三个环境的 `.env` 互相隔离，切换环境时不会读到其它环境的密钥
- [ ] 三种环境的数据库互相隔离，dev 运行不会写入 prod 数据库
- [ ] 文档（README / architecture）中说明如何切换环境
